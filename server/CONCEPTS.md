# Core Concepts

This document explains the design philosophy and key abstractions behind Speculative System Designer.

---

## The Central Problem

Most architecture review processes happen too late and too politely. Teams propose a system, debate it for a few hours, then ship it with unresolved assumptions baked in.

Speculative System Designer forces three things that are usually skipped:

1. **Non-negotiable constraints** declared *before* design begins.
2. **Pessimistic futures** simulated *against* the design.
3. **Explicit tradeoffs** accepted *on the record* before a final architecture is issued.

---

## Roots

Roots are the constitutional layer of the system. They are the constraints you are not willing to violate, stated plainly, before any architecture is generated.

### Structure

```json
{
  "root_id": {
    "id": "root_id",
    "statement": "Plain-language constraint.",
    "rationale": "Why this constraint exists.",
    "violation_examples": [
      "concrete thing that breaks this rule"
    ]
  }
}
```

### The Four Default Roots

| Root | Statement |
|---|---|
| `team_constraints` | The system must be buildable and operable by a team of 6 engineers or fewer. |
| `cost_discipline` | Baseline infrastructure costs must remain predictable and defensible to non-technical stakeholders. |
| `operational_observability` | The system must be diagnosable without requiring ad-hoc instrumentation in production. |
| `reversibility` | Major architectural decisions must be reversible without a full rewrite. |

### How Roots Are Enforced

Roots are injected verbatim into the architecture generation prompt via MCP Sampling. The LLM is instructed to explain how its design satisfies each root before providing the final text. This does not guarantee compliance — but it forces the model to reason about each constraint explicitly rather than ignoring them.

### Adding Your Own Roots

Edit `data/roots.json`. Roots take effect immediately on the next call to `generate_architecture_tool`. They do not require code changes.

## Futures

A future is a pessimistic scenario used to stress-test an architecture after it has been proposed.

The word "future" is deliberate. It is not a "risk" (which implies probability) or a "test" (which implies pass/fail). A future is a world the system might have to operate in — and the question is: *what breaks first?*

### Structure

```json
{
  "future_id": {
    "id": "future_id",
    "description": "What this scenario assumes.",
    "assumption": "The specific pessimistic assumption (e.g. '10x traffic in 18 months').",
    "stance": "pessimistic | adversarial | literal",
    "stress_points": ["..."],
    "failure_modes": ["..."],
    "early_warning_signals": ["..."],
    "review_prompt": "The prompt sent to the LLM. Must instruct the model to return JSON with 'summary' and 'risks'."
  }
}
```

### The Three Default Futures

| Future | Assumption | Stance |
|---|---|---|
| `scaling` | Traffic and data volume increase 10x within 18 months. | Pessimistic |
| `security_abuse` | Attackers are competent and internal misconfigurations will occur. | Adversarial |
| `regulatory_compliance` | New compliance requirements arrive with short notice after the system ships. | Literal |

### The `review_prompt` Contract

The `review_prompt` is sent verbatim to the LLM (via MCP Sampling). It **must** instruct the model to respond only in JSON with two fields:

```json
{
  "summary": "One-paragraph description of what fails and why.",
  "risks": ["Specific risk one", "Specific risk two", "Specific risk three"]
}
```

If the model wraps its response in markdown fences (```json ... ```), the server strips them automatically.

### Adding Your Own Futures

Edit `data/futures.json`. New futures are picked up automatically by `evaluate_architecture_tool` on the next run. No code changes needed.

## Critiques

A Critique is the structured output of running a Future against an Architecture.

```python
class Critique(BaseModel):
    id: str                   # UUID
    future: str               # which future produced this
    summary: str              # what breaks and why
    risks: list[str]          # specific risks (max 3)
    required_tradeoff: str    # "Unresolved" until declared
    resolved: bool            # True once a tradeoff is accepted
```

Critiques are stored in `CRITIQUE_STORE` (in-memory) and referenced by their ID in the architecture's review record inside `REVIEW_STORE`.

---

## Tradeoffs

A Tradeoff is the human's explicit acknowledgement of a compromise.

Every Critique must be resolved by exactly one Tradeoff before a final architecture can be issued. The server enforces this — `finalize_architecture_tool` returns an error if unresolved critiques exist.

Tradeoffs are generated by the LLM (three options per critique) and selected by the human via MCP Elicitation. The selected statement is then:

1. Stored in `REVIEW_STORE[architecture_id]["tradeoffs"]`
2. Set on the `Critique.required_tradeoff` field
3. Used verbatim in the `finalize_architecture_tool` prompt

This means the final architecture is not just "updated" — it is regenerated with the tradeoffs as first-class input, ensuring they actually shape the output.

---

## MCP Sampling vs. Elicitation

Two MCP primitives are central to how authority is split between server and client:

### Sampling (`ctx.session.create_message`)

The **server** sends a prompt to the **client**, which routes it to the client's LLM and returns the result. This keeps LLM calls inside the MCP session without the server needing API keys or model access.

Used for: architecture generation, future simulation, tradeoff option generation, finalization.

### Elicitation (`ctx.elicit`)

The **server** asks the **human** a structured question. The client renders a form (or terminal prompt) and the human's response is returned to the server as validated Pydantic data.

Used for: tradeoff selection (choosing A, B, or C).

This separation is intentional. **The server never lets the LLM choose the tradeoff** — that decision always belongs to the human.

## State Management

All runtime state lives in two in-memory dictionaries:

| Store | Location | Contents |
|---|---|---|
| `REVIEW_STORE` | `server/state/store.py` | One entry per architecture: initial text, tradeoffs, critique IDs, final text |
| `CRITIQUE_STORE` | `server/resources/critiques.py` | One entry per Critique object |

Both stores are reset on server restart. This is appropriate for the current use case (single design sessions in Claude Desktop). For persistent sessions or multi-user deployments, replace the dicts with a database.
